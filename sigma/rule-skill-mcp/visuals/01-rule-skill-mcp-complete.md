# Rule、Skill、MCP 核心区别完全指南

## 一句话总结

| 机制 | 一句话 | 解决的问题 |
|------|--------|-----------|
| **Rule** | 告诉 LLM **怎么想** | LLM 输出质量/风格/约束 |
| **Skill** | 告诉 Agent **怎么做** | Agent 执行流程的复用与自动化 |
| **MCP** | 给 Agent **能做什么** | Agent 的外部系统访问能力边界 |

---

## 三层架构图

```text
┌────────────────────────────────────────────────────────────┐
│                     用户的一次对话请求                       │
└─────────────────────────┬──────────────────────────────────┘
                          │
          ┌───────────────▼──────────────────┐
          │        Rule (Layer 1)            │
          │  静态注入 System Prompt           │
          │  "所有代码用 TypeScript"          │
          │  "回复必须简洁，不超过100字"       │
          │  → 影响 LLM 的推理行为            │
          └───────────────┬──────────────────┘
                          │
          ┌───────────────▼──────────────────┐
          │        Skill (Layer 2)           │
          │  动态行为脚本，驱动 Agent 执行    │
          │  "先读 session.md"               │
          │  "然后出 1-2 道问题"             │
          │  "每轮后更新 roadmap.html"        │
          │  → 影响 Agent 的执行流程          │
          └───────────────┬──────────────────┘
                          │ Skill 调用工具
          ┌───────────────▼──────────────────┐
          │        MCP (Layer 3)             │
          │  运行时能力提供者                 │
          │  github-mcp → 读写 GitHub Issues  │
          │  filesystem-mcp → 读写本地文件    │
          │  browser-mcp → 控制浏览器         │
          │  → 决定 Agent 能访问哪些外部系统  │
          └──────────────────────────────────┘
```

## 核心区别：三个不同的维度

### 维度 1：作用时机

| | 作用时机 |
|---|---|
| Rule | **编译期**（每次发请求前，静态拼入 Prompt） |
| Skill | **运行期**（Agent 执行过程中，动态驱动工具调用） |
| MCP | **启动期**（Agent 环境初始化时，注册可用工具集） |

### 维度 2：影响对象

- **Rule** → 影响的是 LLM（大模型那端）。改变大模型输出什么内容、什么风格。
- **Skill** → 影响的是 Agent Runtime（宿主机那端）。改变 Agent 执行哪些步骤、调用哪些工具。
- **MCP** → 影响的是工具注册表。决定 Agent 有哪些工具可以调用。

### 维度 3：依赖关系

```
MCP 决定了工具注册表（能做什么）
    ↓
Skill 驱动 Agent 按流程调用这些工具（怎么做）
    ↓
Rule 约束 LLM 在每一步推理时的行为（怎么想）
```

**类比**：
- **Rule** = 公司的规章制度（员工的行为准则）
- **Skill** = SOP 标准作业流程（某项工作的具体步骤手册）
- **MCP** = 公司的基础设施（能用哪些系统和工具的权限）

## 选择指南

面对具体需求，应该用哪种机制？

| 需求 | 用哪个 | 理由 |
|------|--------|------|
| "所有代码用 TypeScript" | Rule | 改变 LLM 的输出约束，不需要执行任何操作 |
| "回复时不要用 Emoji" | Rule | 纯输出风格约束 |
| "帮我按照固定流程做 Code Review" | Skill | 需要多步骤执行（读文件、分析、写报告）的可复用流程 |
| "每次我说 /standup，就自动生成日报" | Skill | 有明确触发词、固定执行步骤 |
| "让 Agent 能读 GitHub Issues" | MCP | 访问外部系统的能力扩展 |
| "让 Agent 能查 Slack 消息" | MCP | 访问外部服务的能力扩展 |
| "让 Agent 能用 GitHub MCP 按规范提 PR" | MCP + Skill | MCP 提供能力，Skill 定义流程 |

## 内部结构拆解

### Rule 的内部结构（Cursor `.mdc` 文件）

```
Rule
├── 触发条件（Frontmatter）
│   ├── alwaysApply: true       → 每次对话都注入
│   ├── globs: "**/*.ts"        → 只在打开 .ts 文件时注入
│   └── （手动 @引用）          → 用户主动引用时注入
└── 规则正文（纯文本）
    └── 直接拼进 System Prompt，LLM 当作指令来遵守
```

Rule 没有"执行流程"，因为它不执行任何操作。

---

### Skill 的内部结构（SKILL.md 文件）

```
Skill
├── 元信息（是什么 / 什么时候用）
│   └── 简短描述 + 触发词/指令（如 /sigma）
├── 核心规则（NON-NEGOTIABLE 局部约束）
│   └── 只在本 Skill 激活时生效，不污染其他对话
├── 输入/输出目录声明
│   └── 声明 Skill 会读写哪些文件（如 sigma/{topic}/session.md）
├── 执行流程（分步骤 SOP）
│   ├── Step 0: 解析参数
│   ├── Step 1: 诊断当前状态
│   ├── Step 2: 执行核心逻辑
│   └── Step N: 收尾与更新
└── 完成/推进判断标准
    └── 满足什么条件才算"掌握"（如 mastery >= 80%）
```

**Rule 在 Skill 里的位置**：
- `.cursor/rules/` 的 Rule = **全局 Rule**，作用于所有对话
- Skill 内的核心规则 = **局部 Rule**，仅在本 Skill 执行期间有效

---

### MCP Server 的内部结构

```
MCP Server
├── 传输层（Transport）
│   ├── stdio（本地子进程通信）
│   └── SSE（远程 HTTP 流式通信）
├── tools/list 接口
│   └── 向 Client 暴露工具列表（动态发现）
└── 每个 Tool
    ├── name（工具名称，LLM 用来匹配调用）
    ├── description（工具描述，LLM 靠这个决定用哪个工具）
    ├── inputSchema（JSON Schema，定义参数结构）
    └── execute()（真正操作外部系统的代码）
```

MCP Server 不管"什么时候调用哪个工具"，只管"把工具注册好，等人来调用"。工作流编排是 Skill 的职责。

## 常见误区

❌ **误区 1：Skill 和 Rule 是一回事，都是"给 AI 的指令"**
✅ Rule 只影响 LLM 的文字输出，Skill 会让 Agent 真正执行操作（写文件、运行命令）。

❌ **误区 2：MCP 是 Skill 的增强版**
✅ MCP 和 Skill 是完全不同的维度：MCP 决定"能做什么"，Skill 决定"怎么做"。没有 MCP，Skill 中调用 GitHub 的指令会静默失败。没有 Skill，MCP 只是把工具暴露出来，Agent 不会主动按照业务流程去用它们。

❌ **误区 3：有了 MCP 就不需要 Skill**
✅ MCP 只是"工具注册"。Skill 是"工作流编排"。前者是基础设施，后者是业务逻辑。
