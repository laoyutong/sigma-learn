# 01. Agent 与系统的物理边界 (Host vs Target)

## 核心概念：大脑与手脚的分离

在 AI Agent 架构中，最容易让人产生错觉的就是“AI 帮我操作了电脑”。实际上，AI（大模型）本身**没有任何执行能力**，它只是一个纯粹的文本/Token 处理器。

必须在架构上严格区分三个角色：

1. **大脑 (LLM)**：负责分析意图、规划步骤、决定调用什么工具、生成工具的参数。它通常运行在云端（如 OpenAI 服务器），完全隔离于你的本地环境。
2. **手脚 (Host/Agent Runtime)**：运行在你本地电脑或服务器上的代码（比如你用 TypeScript 写的 LangChain 脚本）。它负责与 LLM 通信，接收 LLM 的指令，并在本地**真正执行**代码逻辑（如 `fs.readFileSync`）。
3. **操作对象 (Target)**：被 Host 操作的系统或资源，比如本地文件系统、数据库、外部 API 等。

## ASCII 架构图解

```text
+---------------------+         网络请求         +-----------------------+
|    云端大模型 (LLM) | <----------------------> |    本地宿主机 (Host)  |
|                     |   (Prompt / Tool Call)   |  (运行 Agent 的 Node) |
|  - 意图识别         |                          |  - 管理对话历史       |
|  - 决定调用工具     |                          |  - 执行工具函数代码   |
|  - 生成参数         |                          |  - 处理错误与重试     |
+---------------------+                          +-----------+-----------+
                                                             |
                                                             | 执行 fs.readFile()
                                                             v
                                                 +-----------------------+
                                                 |    目标系统 (Target)  |
                                                 |  (本地操作系统/文件)  |
                                                 |  - ~/.ssh/id_rsa      |
                                                 |  - 数据库             |
                                                 +-----------------------+
```

## 认知误区纠正

- ❌ **误区**：GPT-4 黑进了我的电脑，读取了我的私钥。
- ✅ **真相**：GPT-4 只是返回了一段类似 `{"name": "read_file", "args": {"path": "~/.ssh/id_rsa"}}` 的 JSON 文本。是你本地运行的 Agent 代码（Host）解析了这个 JSON，并且**你的代码**执行了读取文件的操作。如果你的代码没有写 `fs.readFile`，大模型再怎么想读也读不到。

## 关键结论

**大模型只负责“想”，宿主机才负责“做”。**
理解了这个物理边界，我们就明白了为什么需要**工具调用协议（Tool Calling Protocol）**——因为大脑和手脚是断开的，它们需要一种标准化的语言来沟通“做什么”和“怎么做”。
