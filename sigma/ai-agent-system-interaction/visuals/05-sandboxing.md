# 05. 沙箱与安全隔离 (Sandboxing)

## 为什么需要沙箱？

虽然我们有“二次确认”（Human-in-the-loop）机制，但这依赖于人类的警惕性。如果用户在疲惫时习惯性地点击了“允许”，或者 Agent 运行在无人值守的服务器上处理批量任务，二次确认就失效了。

为了防止不可逆的破坏，我们需要在操作系统层面建立一道物理/虚拟的隔离墙，这就是**沙箱（Sandbox）**。

## 沙箱的核心目标

沙箱的本质是**限制 Agent 进程的系统调用（Syscalls）权限**。
1. **文件系统隔离**：Agent 只能看到和修改一个虚拟的文件系统，或者被限制在一个特定的目录（如 Workspace）内，无法读取 `/etc/passwd` 或用户的私钥。
2. **网络隔离**：限制 Agent 只能访问特定的白名单域名（如 `github.com`, `npmjs.org`），防止它把敏感数据发送到黑客的服务器，或者扫描内网。
3. **进程与资源限制**：防止 Agent 启动挖矿程序耗尽 CPU，或者触发 Fork 炸弹。

## 常见的沙箱技术方案

在 AI Agent 领域，目前主流的沙箱技术有以下几种：

### 1. 容器化隔离 (Docker / 虚拟机)
- **原理**：为 Agent 启动一个轻量级的 Linux 容器（如 Docker）或微虚拟机（如 Firecracker）。Agent 的所有操作都在这个虚拟环境中进行。
- **优点**：隔离彻底，Agent 可以自由执行各种 Shell 命令（甚至 `rm -rf /`），因为破坏的只是一个阅后即焚的容器。
- **缺点**：启动慢，资源消耗大。

### 2. WASM (WebAssembly)
- **原理**：将 Agent 的执行环境编译成 WASM 字节码，运行在一个严格受限的 WASM 运行时（如 Wasmtime）中。
- **优点**：启动极快（毫秒级），安全性极高（WASM 默认没有任何系统权限，必须通过 WASI 显式授权）。
- **应用**：很多云端 Agent 平台使用 WASM 来安全地执行大模型生成的代码。

### 3. 操作系统级沙箱 (macOS Sandbox / Linux seccomp)
- **原理**：利用操作系统底层的机制拦截系统调用。例如，Cursor 在执行某些命令时，会使用 macOS 的 `sandbox-exec` 或 Linux 的 `seccomp` 配置文件。
- **特点**：不需要启动虚拟机，直接在宿主机运行，但通过 OS 内核限制其行为。如果 Agent 尝试读取白名单之外的文件，系统调用会直接返回 `Permission Denied`。

## 总结：防御纵深

一个安全的 AI Agent 系统，其防御体系是多层的（Defense in Depth）：
1. **Prompt 层面**：告诉大模型不要做坏事（最脆弱）。
2. **Host 层面**：静态作用域限制 + 高危操作的二次确认（Human-in-the-loop）。
3. **OS 层面**：沙箱隔离（Docker/WASM/seccomp），即使代码失控，也无法造成真实破坏（最坚固）。
