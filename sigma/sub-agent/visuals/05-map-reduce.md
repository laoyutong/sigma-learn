# 并行 vs 串行与结果聚合 (Map-Reduce 模式)

## 粗暴并行的灾难：上下文断裂

当让 5 个 Writer Agent 完全并行写一篇文章的 5 个章节时，会发生：
1. **内容割裂**：第二章（寒冬）的开头无法承接第一章（起源）的结尾。
2. **重复冗余**：第三章和第四章可能都花大量篇幅解释了什么是神经网络。
3. **风格不一**：有的幽默，有的学术。

这就是**没有前后联动关系**导致的必然结果。

## 架构解法：Map-Reduce (分散-聚合) 模式

你提出的"并行生成一版之后，进行一个统一的优化改造"直指核心。在 Sub-Agent 架构中，这被称为 **Map-Reduce 模式**（借用了大数据的概念）。

为了既保证速度，又保证质量，我们需要引入两个关键机制：**全局大纲（Skeleton）** 和 **聚合优化（Reducer）**。

### 完整工作流推演

```
[ 用户请求: 写万字长文 ]
         │
         ▼
┌─────────────────────────────────────────────────────────┐
│ 1. Planner Agent (生成骨架 Skeleton)                    │
│ 输出: 详细的大纲，明确规定每章写什么、不写什么、字数限制。│
└────────────────────────┬────────────────────────────────┘
                         │
        并行 Map 阶段 (带着骨架约束)
    ┌──────────┬─────────┼─────────┬──────────┐
    ▼          ▼         ▼         ▼          ▼
┌───────┐  ┌───────┐ ┌───────┐ ┌───────┐  ┌───────┐
│Writer1│  │Writer2│ │Writer3│ │Writer4│  │Writer5│
│ 第一章 │  │ 第二章 │ │ 第三章 │ │ 第四章 │  │ 第五章 │
└───────┘  └───────┘ └───────┘ └───────┘  └───────┘
    │          │         │         │          │
    └──────────┴─────────┼─────────┴──────────┘
                         ▼
┌─────────────────────────────────────────────────────────┐
│ 2. Reducer Agent (也就是你说的"统一优化改造")           │
│ 职责: 拼接、润色过渡段、消除重复、统一语气。            │
└────────────────────────┬────────────────────────────────┘
                         │
                         ▼
                   [ 最终连贯的长文 ]
```

### TypeScript 伪代码实现

```typescript
async function generateLongArticle(topic: string) {
  // 1. 串行：生成强制骨架（Skeleton）
  const outline = await plannerAgent.generate(`
    为主题"${topic}"生成一个 5 章的详细大纲。
    必须明确指出每章的边界，防止内容重叠。
  `);
  
  // 2. 并行 (Map)：带着骨架去写
  // 关键：把整个大纲都传给每个 Writer，让它们知道自己的"上下文位置"
  const chapterPromises = outline.chapters.map(chapter => {
    const prompt = `
      全局大纲是：${JSON.stringify(outline)}
      你的任务是只写第 ${chapter.index} 章。
      注意不要写其他章节的内容。
    `;
    return writerAgent.write(prompt);
  });
  
  const rawChapters = await Promise.all(chapterPromises);
  
  // 3. 串行 (Reduce)：统一优化
  const finalArticle = await reducerAgent.optimize(`
    这里是 5 个独立生成的章节：${rawChapters.join("\n\n")}
    请你：
    1. 为每两章之间增加平滑的过渡段
    2. 统一整篇文章的语气
    3. 删掉重复解释的概念
  `);
  
  return finalArticle;
}
```

## 核心原则：何时串行，何时并行？

| 模式 | 适用场景 | 架构代价 |
|------|----------|----------|
| **纯串行 (Chain)** | 强因果依赖（如：先查数据，再写报告） | 速度慢，延迟高 |
| **纯并行 (Parallel)** | 任务完全独立（如：查三家不同公司的财报） | 速度快，但无法共享中间状态 |
| **Map-Reduce** | 任务可拆分，但最终需要强一致性（如：写长文、总结长视频） | 需要多消耗 Token 进行 Reducer 的二次处理 |

**架构师的直觉**：在 Sub-Agent 系统中，尽可能把任务转化为 Map-Reduce 模式。用 Planner 建立全局约束（Map），用 Reducer 解决局部割裂（Reduce）。
