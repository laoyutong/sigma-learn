# 主流 Sub-Agent 拓扑模式对比

在多 Agent 系统中，Agent 之间的组织结构（拓扑）决定了系统的灵活性、可控性和开发成本。业界目前有三种主流模式。

## 1. Supervisor (主管模式 / 星型拓扑)
这是最常见、最容易控制的模式。

```
       [ 用户 ]
          │
    ┌─────▼─────┐
    │ Supervisor│ (中心路由)
    └─┬───┬───┬─┘
      │   │   │
   ┌──▼┐ ┌▼──┐ ┌▼──┐
   │ W1│ │ W2│ │ W3│
   └───┘ └───┘ └───┘
```
- **特点**：Worker 之间互不相见，所有信息必须经过 Supervisor 中转。
- **优点**：状态管理极度清晰，容错容易做（Supervisor 捕获所有错误）。
- **缺点**：Supervisor 容易成为瓶颈（Token 消耗大，上下文容易溢出）。
- **适用场景**：任务边界清晰、需要强聚合的场景（如：写行业分析报告）。

## 2. Hierarchical (层级模式 / 树状拓扑)
Supervisor 模式的嵌套版本。

```
          [ CTO Agent ]
          /           \
   [ 前端主管 ]      [ 后端主管 ]
    /       \          /      \
[React]   [CSS]    [DB]    [API]
```
- **特点**：每一层只负责向下拆解任务和向上汇报结果。
- **优点**：无限扩展能力，完美解决单一 Supervisor 上下文溢出的问题。
- **缺点**：延迟极高（信息传递链路太长），开发和调试成本巨大。
- **适用场景**：超大型、超长周期的任务（如：从零开发一个完整的 App）。

## 3. Swarm (蜂群模式 / 网状拓扑 / Handoff)
这是 OpenAI 最近开源的 `Swarm` 框架主推的模式，也是你选择的模式。

```
[ 用户 ] ──> [ Agent A (读代码) ]
                   │
             (Handoff: 移交控制权)
                   ▼
             [ Agent B (写代码) ]
                   │
             (Handoff: 移交控制权)
                   ▼
             [ Agent C (跑测试) ] ──> (如果失败，移交回 Agent B)
```
- **特点**：没有中心老板。Agent 通过特殊的 `handoff` (移交) 工具，把对话的控制权直接交给下一个 Agent。
- **优点**：
  1. **极度灵活**：流程可以像状态机一样自由流转（比如 B -> C -> B -> C 循环）。
  2. **上下文无缝继承**：移交时，下一个 Agent 直接继承当前的对话历史。
- **缺点**：容易陷入死循环（A 踢给 B，B 踢给 A），全局状态难以追踪（不知道现在到底在谁手里）。
- **适用场景**：流程偏串行、需要频繁交接上下文、步骤不确定的探索性任务（如：**修 Bug、客服路由**）。

## 为什么修 Bug 适合 Swarm 模式？

你的直觉非常准。修 Bug 的过程本质上是一个**探索性的串行状态机**：
1. `ReaderAgent` 找代码，找到后觉得需要修改，直接把上下文 **Handoff** 给 `WriterAgent`。
2. `WriterAgent` 改完代码，直接 **Handoff** 给 `TestAgent`。
3. `TestAgent` 跑测试报错了，它不需要汇报给什么"主管"，而是直接带着错误信息 **Handoff** 回给 `WriterAgent`。

如果用 Supervisor 模式，每次 `Writer` 和 `Test` 之间的交互都要经过 `Supervisor` 中转，不仅浪费 Token，而且 `Supervisor` 根本不需要关心这些底层的编译错误。Swarm 模式让专业的人直接对接专业的人，效率最高。
